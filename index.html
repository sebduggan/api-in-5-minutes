Back in the early days of the web - when I built my first site, 25 years ago - sites were mostly isolated and self-contained.

Servers didn't talk to each other, and the browsers of the day - REVEAL - certainly did not have the capability of accessing data on the client side.

Stand-alone websites

⇓

Data feeds over FTP

⇓

WSDL, SOAP XML etc.

⇓

REST APIs

Over the years, things have improved, and it is a basic assumption now that a website will communicate data with one or more other sites, either via the server or on the client.

And the driving force behind that is REST, which is now widely used both for server-to-server data communications, and for the client-server interactions required both by modern websites and mobile apps.

SaaS applications

Web apps

Embedded widgets

Mobile apps

  • SaaS: CRM sync, raising invoices, etc
  • Web apps:
  • Widgets: Twitter etc
  • Mobile apps need a simple way to transfer data
  • Preside is very data-centric
  • Many clients require syncing of data to/from external systems
  • More and more, we're required to make data available from Preside; e.g. exporting event attendees to conference apps, or exposing our CRM data to external systems
  • Preside has built-in REST capabilities, handling requests and routing
  • Made sense to leverage the existing data model to make the process of setting up an API as easy as possible...

Preside’s approach to REST

The Preside Data API uses four verbs:

  • GET
  • POST
  • PUT
  • DELETE

PATCH is not used

All updates use PUT

Only supplied fields are updated

Response body contains only the data

Metadata is passed in the headers

API is documented separately

No to HATEOAS Hypertext As The Engine Of Application State

REST responses

200: Success

401: Authorization failed

404: Not found

405: Method not supported

422: Validation failure

500: Server error

Data change queues

  • By default, psermissions off for users
  • Enable data change queue per object, per operation
  • Here, just enable all operations, all objects
/config/Config.cfc 

settings.rest.apis[ "/data/v1" ].dataApiQueueEnabled = true;
settings.rest.apis[ "/data/v1" ].dataApiQueues = {
    default     = { pageSize=100, atomicChanges=true  }
  , lowpriority = { pageSize=10 , atomicChanges=false }
};
  • By default, queue page size is 1 record.
  • Atomic - each individual change is queued in order.
  • Multiple queues - critical data in default, checked constantly. Less important data in "lowpriority" queue - check less frequently
  • Use @dataQueueName annotation on preside object

Authentication

Basic authentication

Users manually created in admin

Access token passed in as username

Potential for self-service by website users

Please use HTTPS!

  • Programmatic creation of API user
  • Business logic defines permissions
  • Show the user their access token
  • Allow them to reset their token

No authentication

WARNING!
Only use for a read-only API

/config/Config.cfc 

settings.rest.apis[ "/data/v1" ].authProvider = "";

Could use for a firewall-protected or internal application, but even so should probably use Basic auth!
